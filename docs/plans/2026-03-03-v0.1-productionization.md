# Converge v0.1 Productionization (README + Packaging + CI/CD)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows `PLANS.md` at the repository root.

## Purpose / Big Picture

Converge already has core local-first experiment tracking capabilities, but onboarding and release ergonomics are not production-ready. After this work, a new user can install Converge with Homebrew in one command, run a short quickstart successfully, and trust tagged releases because CI and release automation are enforced.

## Surprises & Discoveries

- Observation: the repository remote and Go module path were inconsistent (`prit3010` remote vs `prittamravi` module path).
  Evidence: `git remote -v` returned `git@github.com:prit3010/converge.git` while `go.mod` used `github.com/prittamravi/converge`.
- Observation: GoReleaser now recommends Cask-based Homebrew distribution and deprecates the old `brews` integration.
  Evidence: GoReleaser deprecation note for `brews` in favor of `homebrew_casks`.
- Observation: Existing `.gitignore` patterns (`docs/` and `converge`) blocked tracking both new docs and `cmd/converge/main.go`.
  Evidence: `git status --short` originally did not surface new docs or the `cmd/converge` entrypoint until ignore patterns were scoped.
- Observation: Local execution environment does not include `go`/`gofmt`, so compile-time validation could not be completed here.
  Evidence: `go test ./...`, `go build ./...`, and `gofmt -w ...` each returned command-not-found errors.

## Context and Orientation

Converge is a Go CLI with command wiring in `internal/cli/`, business logic in `internal/core/`, and entrypoint in `cmd/converge/main.go`. At the start of this effort, user-facing documentation lived almost entirely in `README.md`, and there was no `.github/workflows/` directory, release automation config, or explicit license file.

The productionization scope is intentionally limited to release-readiness surfaces:

- onboarding documentation and quickstart clarity,
- packaging/release automation (GitHub Releases + Homebrew Cask tap),
- continuous integration quality gates,
- one small CLI enhancement (`version`) for install verification.

No changes to core snapshot/eval/storage behavior are expected.

## Plan of Work

First, normalize repository identity and CLI release metadata. Update module/import paths to `github.com/prit3010/converge`, then add `internal/cli/version.go` and wire it in `internal/cli/root.go`, plus tests in `internal/cli/version_test.go`.

Second, rewrite top-level docs to optimize first-run success. Replace `README.md` with a concise install-first guide and add focused docs at `docs/install.md`, `docs/quickstart.md`, `docs/hooks.md`, and `docs/releasing.md`. Keep advanced details discoverable via links rather than inline bulk text.

Third, add release governance. Create `.github/workflows/ci.yml` for required checks and `.github/workflows/release.yml` for tag-triggered releases. Add `.goreleaser.yaml` with multi-platform binary builds, checksum generation, ldflags for metadata injection, and `homebrew_casks` configuration to update `prit3010/homebrew-converge`.

Finally, add `LICENSE` (MIT), then run local checks as available in environment.

## Validation and Acceptance

Acceptance is complete when:

- `converge version` prints version metadata in human output and `--json` envelope output.
- README starts with install instructions and points to dedicated docs for deeper workflows.
- CI workflow executes formatting, tests, vet/build, and race tests.
- Release workflow triggers on tags and calls GoReleaser.
- GoReleaser config produces archives/checksums and includes Homebrew Cask automation for `prit3010/homebrew-converge`.

## Idempotence and Recovery

All doc/config edits are idempotent. Re-running CI/release workflows is safe because release publishing is tag-scoped. If a release fails mid-flight, fix config and re-run workflow for the same tag after cleaning draft artifacts. If module-path migration causes build issues in external consumers, temporary compatibility can be restored by pinning to pre-v0.1 tags.
